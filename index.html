Boot-up Flow

On a Qualcomm-based Android Automotive device, boot is not a single process — it is a chained handoff across eight distinct stages, each verifying and loading the next. Understanding where your code fits in this chain is fundamental to debugging any early-boot issue.

Architect Insight

On automotive platforms (SA8155P), the rear camera must be visible within 2 seconds of ignition. This hard constraint drives early Venus (video) firmware loading and a dedicated fast-boot path for the display/camera pipeline, often bypassing full Android init.

02

Kernel to Init Flow

The question "how does the kernel start init when init lives in /system?" reveals a critical two-stage architecture. The kernel never touches the system partition directly. It executes /init from an in-RAM filesystem built from the ramdisk in boot.img.

⚠️
Common Misconception

The kernel does not mount /system to find init. When the kernel first runs /init, /system does not exist yet. The /init binary comes from a RAM-based tmpfs built from the boot.img ramdisk.

Deep Dive Topics

What is populate_rootfs() and how does it work?

populate_rootfs() is registered as a rootfs_initcall in the kernel, meaning it runs before any userspace process starts. Its job: decompress the cpio ramdisk blob placed in RAM by ABL, and extract every entry into the tmpfs already mounted at /.

C — init/initramfs.c (simplified)
static int __init populate_rootfs(void)
{
    /* 1. Unpack built-in initramfs (kernel embedded cpio) */
    unpack_to_rootfs(__initramfs_start, __initramfs_size);

    /* 2. Unpack external ramdisk (from ABL via DTB chosen node) */
    if (initrd_start) {
        unpack_to_rootfs((char *)initrd_start,
                          initrd_end - initrd_start);
        free_initrd(); /* release RAM once extracted */
    }
    return 0;
}
rootfs_initcall(populate_rootfs); /* register at boot priority */

The function runs twice: once for the built-in initramfs embedded in the kernel binary, and once for the external ramdisk loaded by ABL. On Android 11+, ABL merges the generic ramdisk (from boot.img) and vendor ramdisk (from vendor_boot.img) into a single cpio blob before passing it to the kernel.

Why does Android NOT use switch_root?

Traditional Linux systems use switch_root to pivot from initramfs to a real block device filesystem. Android explicitly avoids this for two reasons:

1. Root stays as tmpfs forever — / remains the in-memory tmpfs throughout device lifetime. Real partitions (/system, /vendor) are mounted as subdirectories inside that tmpfs root, not as a replacement root.

2. PID 1 must remain PID 1 — switch_root would create a new process. Instead, Android uses execv() which replaces the process image in-place, keeping the same PID. This is critical because the kernel communicates SIGCHLD for orphaned processes to PID 1.

ℹ️
Key Fact

The three execv calls (kernel→first-stage, first-stage→selinux_setup, selinux_setup→second_stage) all keep PID 1. The binary changes but the process does not.

init.rc — Actions, Services, and Triggers

Second stage init parses .rc files from /system/etc/init/, /vendor/etc/init/, and /odm/etc/init/. The syntax has three main constructs:

init.rc — annotated example
# ACTION — runs when trigger fires
on early-init
    mount_all /vendor/etc/fstab.qcom --early
    # mounts persist, metadata early

on init
    setprop ro.hardware qcom

on property:sys.boot_completed=1
    # runs when ANY service sets this prop
    start vendor.qti.hardware.wifi

# SERVICE — defines a long-running daemon
service zygote /system/bin/app_process64
    class main
    socket zygote stream 660 root system
    onrestart restart audioserver
    # socket creates /dev/socket/zygote

service vendor.qti.display-hal /vendor/bin/hw/android.hardware.graphics.composer
    class hal
    user system
    group graphics
    capabilities SYS_NICE

Service classes define start order: core → hal → main. Class hal starts before main, which means HAL services are available before Zygote starts.

Android Verified Boot (AVB)

AVB 2.0 is a layered, cryptographic chain of trust from hardware fuses to runtime block verification. Every layer verifies the next. A break anywhere in the chain prevents the device from booting in green state.

Block Diagram — AVB Chain of Trust

Fig 3.1 — AVB chain of trust. Each layer cryptographically verifies the next. Hardware eFuse is the immutable root. dm-verity provides runtime protection after boot completes.

RPMB — Physical Location and How It Works

RPMB (Replay Protected Memory Block) is a dedicated hardware partition physically inside the eMMC or UFS storage controller — not a separate chip. It enforces HMAC-SHA256 authentication on every read and write.

RPMB Write Flow (Simplified)
/* TrustZone (EL3) only — no HLOS direct access */

1. Request write counter from RPMB
   eMMC → { counter=42, HMAC(key, counter) }
   TZ verifies HMAC

2. Prepare write:
   data = { new_rollback_value, address, counter+1 }
   MAC  = HMAC-SHA256(RPMB_KEY, data)
   Send { data, MAC } to eMMC

3. eMMC validates MAC + counter increment
   ✅ accept → write + increment counter (irreversible)
   ❌ reject → no change

The RPMB authentication key is derived by TrustZone from the SoC's HUK (Hardware Unique Key) — fused into the Qualcomm chip, never extractable. Even physical replacement of the eMMC chip resets rollback indices to zero, which is why Qualcomm platforms add eFuse-based version counters as a second line of defense.

Recovery Flow

When you run adb reboot recovery, a chain of events spans from the ADB protocol layer down to a BCB (Bootloader Control Block) write in the misc partition — the only communication channel that survives a SoC reset. The bootloader reads it after reboot to choose recovery over Android.

Block Diagram — Recovery Decision Architecture

Fig 4.1 — Recovery flow. The BCB in misc partition is the key: written by Android init before reboot, read by ABL after reset. The misc partition is the only durable, non-volatile communication channel between the Android OS and the bootloader.

Fig 4.2 — Complete adb reboot recovery sequence. Key insight: BCB persists across the SoC reset (it is in eMMC/UFS misc partition, not in RAM). After recovery completes, it clears BCB to ensure next boot is normal.

VAB Recovery vs Dedicated Recovery Partition

Dedicated recovery (Android ≤10): A separate recovery partition contains its own kernel, ramdisk, and recovery binary. ABL loads this entire separate image when recovery is needed. Update: this partition takes permanent storage space even when not in use.

VAB recovery (Android 11+): No dedicated recovery partition. The same boot.img kernel is used. ABL sets androidboot.force_normal_boot=0 in the kernel cmdline. First stage init reads this flag and takes the recovery code path — execing /system/bin/recovery instead of /system/bin/init.

💡
Debugging Tip

If a device reboots into recovery in a loop, first check the BCB: adb shell dd if=/dev/block/by-name/misc bs=1 count=32 | strings. If boot-recovery appears, something is writing the BCB (possibly a failed OTA update).

HLOS & Non-HLOS Images

A Qualcomm SoC is not one computer — it is 8–10 independent processors on a single die, each running its own firmware or OS. HLOS (Android/Linux) is just one of them. The others are NON-HLOS subsystems, each with hard real-time requirements that Linux cannot satisfy.

FAQ

Frequently Asked Questions

Q1 Why does init run twice (selinux_setup then second_stage) instead of once? ▼

SELinux policy must be loaded before any security-sensitive process starts, but the policy files live in /system/etc/selinux/ — which is only accessible after /system is mounted.

First stage init mounts /system, then execvs to /system/bin/init selinux_setup. This phase loads and enforces SELinux. Once SELinux is active, it execvs again to second_stage which starts all services. This staged approach ensures every service started by init already runs under SELinux enforcement — there is no window where services could start without policy.

The three-step execv (first-stage → selinux_setup → second_stage) preserves PID 1 throughout while meeting the ordering constraint: mount partitions first, then enforce security, then start services.

Q2 Can you mount /system as read-write and keep dm-verity enabled? ▼

No. dm-verity is fundamentally incompatible with writable mounts. dm-verity works by reading a block, computing its SHA-256, and comparing it against the Merkle hash tree. If you write to a block, its hash changes immediately — the next read of that block would compute a different hash and fail verification, causing an I/O error or device restart.

This is why /system is always mounted MS_RDONLY. For development work, you have two options: (1) unlock the bootloader (fastboot oem unlock) which disables AVB and dm-verity, or (2) use adb remount on a debug build which disables dm-verity for that session via disable-verity in vbmeta. Neither is acceptable for production automotive builds.

Q3 What is the difference between vendor_dlkm and system_dlkm? ▼

vendor_dlkm contains kernel modules owned and signed by the SoC vendor (Qualcomm). These are hardware-specific drivers: display, audio DSP interface, GPU, camera pipeline, CAN bus. Qualcomm ships these as .ko files compiled against the stable KMI (Kernel Module Interface). They are verified by vbmeta_vendor signed with Qualcomm's key.

system_dlkm contains kernel modules owned and signed by Google. These are generic framework-level modules: fuse.ko, incremental_fs.ko (for Play Store streaming installs), dm_user.ko. They are verified by vbmeta_system signed with Google's key.

The separation reflects organizational ownership — each party signs their own modules, can update them independently, and there is no key sharing between Google and Qualcomm.

Q4 Why does persist never get wiped on factory reset, and what happens if you do wipe it? ▼

The persist partition stores factory-measured hardware calibration data: WiFi RF calibration offsets, Bluetooth NV items, display panel color LUT, gyroscope zero-offset, and device-specific sensor calibration. This data is measured on each individual device using specialized factory equipment that accounts for manufacturing variance between units.

Wiping persist is catastrophic because: (1) WiFi transmits at incorrect power levels — a regulatory violation. (2) Display colors are wrong — uncalibrated panel. (3) Sensors drift — IMU zero-offset is lost. This data cannot be recalibrated without returning the device to factory test equipment. The frp partition has similar protection but for anti-theft reasons (Google account binding), not hardware calibration.

Recovery explicitly skips persist when wiping. If you run fastboot erase persist, the device may still boot but radio and sensor functionality will be degraded or regulatory non-compliant.

Q5 How does GKI allow Google to ship kernel security patches without OEM involvement? ▼

Before GKI (Android 10 and earlier), each OEM shipped a monolithic kernel with Qualcomm drivers compiled directly into it. Patching a kernel CVE required: Google patches upstream → Qualcomm backports to their BSP → OEM integrates → tests full regression → ships OTA. This took 6–12 months, leaving devices vulnerable.

GKI separates the kernel into two parts: (1) the GKI core kernel in the boot partition, maintained by Google, with a stable KMI (Kernel Module Interface). (2) vendor kernel modules in vendor_dlkm, compiled by Qualcomm against the stable KMI. Because the KMI is stable across a GKI version, Qualcomm's .ko files continue working when Google ships a patched kernel.

The result: Google can update the boot partition kernel via a Mainline module update or GKI security patch in days to weeks without requiring Qualcomm to recompile or OEMs to run full regression. The vendor_dlkm modules are unaffected as long as KMI is maintained.
